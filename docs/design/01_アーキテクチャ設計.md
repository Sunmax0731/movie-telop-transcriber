# アーキテクチャ設計

## 1. 目的
アプリ全体の責務分離、モジュール構成、データフロー、ディレクトリ構成の大枠を定義する。

## 2. システム全体構成
本システムは、Windows 上で動作する GUI アプリ本体と、動画解析処理群から構成する。

主要構成要素:
1. GUI アプリ本体
2. 動画入力・メタデータ取得層
3. フレーム抽出層
4. OCR ワーカー層
5. 属性解析層
6. テロップ統合層
7. 出力層
8. ログ・作業ディレクトリ管理層

## 3. アーキテクチャ方針
### 3.1 基本方針
- GUI と解析処理を責務分離する。
- OCR エンジンへの依存はワーカー層へ閉じ込める。
- フレーム抽出、OCR、属性解析、統合、出力を段階的に再実行できる構成にする。
- JSON を正本、CSV を派生出力とする。

### 3.2 採用イメージ
- フロントエンド: WinUI 3
- アプリケーション制御: C# 側のオーケストレーション
- OCR ワーカー: ローカル実行コンポーネント
- 永続化: ローカルファイルベース

## 4. モジュール構成
### 4.1 GUI アプリ本体
責務:
- 動画選択
- 設定入力
- 実行開始 / 停止
- 進捗表示
- タイムライン表示
- 結果参照
- サブ画面管理

入力:
- 利用者操作
- 処理進捗
- 解析結果

出力:
- 実行要求
- 設定情報
- 保存要求

### 4.2 動画入力・メタデータ取得層
責務:
- 動画ファイルの存在確認
- 対応形式候補確認
- デコード可否確認
- 動画メタデータ取得

出力:
- `source_video` 情報
- 読み込み可否結果

### 4.3 フレーム抽出層
責務:
- 時刻基準でフレーム抽出を行う
- 抽出画像を作業ディレクトリへ保存する
- フレーム一覧を生成する

出力:
- フレーム画像群
- フレーム単位メタデータ

### 4.4 OCR ワーカー層
責務:
- フレーム画像から文字領域検出を行う
- OCR により文字列を取得する
- OCR 結果を共通形式へ正規化する

入出力方針:
- GUI 本体は OCR エンジン固有 API を直接知らない
- ワーカー層は共通の入出力契約を返す

### 4.5 属性解析層
責務:
- 文字色、枠色、地色を推定する
- フォント分類、サイズ推定を行う
- `text_type` の候補を付与する

出力:
- フレーム単位の属性付き検出結果

### 4.6 テロップ統合層
責務:
- 連続フレーム上の同一テロップを統合する
- セグメント単位の開始時刻、終了時刻、代表属性を確定する

出力:
- セグメント単位結果

### 4.7 出力層
責務:
- JSON 出力
- CSV 出力
- 実行ログ要約出力

### 4.8 ログ・作業ディレクトリ管理層
責務:
- 作業ディレクトリの作成
- フレーム画像、ログ、出力ファイルの配置
- 再実行時の既存成果物との整合確認

## 5. データフロー
1. GUI が動画ファイルと解析設定を受け取る。
2. 動画入力層が入力を検証し、メタデータを返す。
3. フレーム抽出層が時刻基準でフレーム画像を生成する。
4. OCR ワーカー層がフレーム単位の文字検出と文字認識を行う。
5. 属性解析層が色、フォント、サイズ、種別候補を付与する。
6. テロップ統合層がセグメント単位結果へ統合する。
7. 出力層が JSON / CSV / ログを保存する。
8. GUI が進捗と結果を表示する。

GUI 内の表示連動:
1. セグメント結果を受け取ったらタイムライン表示モデルを生成する。
2. タイムライン選択時は、該当セグメントを結果一覧とプレビューへ反映する。
3. 結果一覧選択時は、タイムライン上の該当区間をハイライトする。
4. プレビュー時刻移動時は、タイムラインカーソル位置を更新する。

## 6. データ受け渡し方針
### 6.1 層間の基本ルール
- 各層は入力と出力を明確に分ける。
- OCR エンジン固有の生データは、ワーカー層外へ漏らさない。
- フレーム単位結果とセグメント単位結果は明確に分離する。

### 6.2 主要な受け渡し単位
- `VideoSource`
  - 入力動画情報
- `FrameRecord`
  - 抽出フレーム情報
- `DetectionRecord`
  - OCR と属性解析を含むフレーム単位検出情報
- `SegmentRecord`
  - 統合済みテロップ情報
- `TimelineSegmentViewModel`
  - タイムライン表示用のセグメント情報
- `ExportPackage`
  - JSON / CSV / ログ出力対象

### 6.3 タイムライン表示モデル方針
- タイムライン表示は `SegmentRecord` から生成する。
- 表示に必要な最小属性は以下とする。
  - `segment_id`
  - `start_timestamp_ms`
  - `end_timestamp_ms`
  - `text`
  - `text_type`
  - `selected`
- GUI 表示専用の状態は永続データモデルへ混在させない。

## 7. 非機能設計の基本方針
### 7.1 オフライン性
- 主要処理はローカルで完結する。
- 外部 API 呼び出しを必須にしない。

### 7.2 再実行性
- フレーム抽出以降の工程を段階的にやり直せる余地を持たせる。
- 中間成果物を作業ディレクトリへ保存する。

### 7.3 保守性
- GUI、OCR、属性解析、統合、出力の責務を分ける。
- 将来の OCR 差し替えを可能にする。

### 7.4 ログ性
- 入力条件
- 抽出設定
- OCR エンジン名
- 処理時間
- 警告件数
- エラー件数

を最低限ログ化する。

## 8. ディレクトリ構成の基本方針
- `src/`
  - アプリ本体と処理実装
- `docs/`
  - 要件、仕様、設計、テスト、運用文書
- `work/`
  - 実行時作業領域
- `out/`
  - 出力成果物

### 8.1 実行時ディレクトリ構成案
`work/` 配下:
- `work/runs/<run_id>/frames/`
- `work/runs/<run_id>/ocr/`
- `work/runs/<run_id>/attributes/`
- `work/runs/<run_id>/logs/`

`out/` 配下:
- `out/<run_id>/segments.json`
- `out/<run_id>/segments.csv`
- `out/<run_id>/frames.csv`
- `out/<run_id>/summary.log`

### 8.2 配置方針
- 実行ごとに `run_id` 単位でディレクトリを分ける。
- 中間成果物と最終成果物を分離する。
- JSON を正本、CSV を派生物として扱う。
- ログは中間成果物側と最終成果物側の両方から追跡できるようにする。

### 8.3 クリーンアップ方針
- `work/runs/<run_id>/` は再実行や検証のため保持可能とする。
- GUI から実行単位で削除できる構成を優先する。
- `out/<run_id>/` は成果物として明示的削除以外では保持する。
