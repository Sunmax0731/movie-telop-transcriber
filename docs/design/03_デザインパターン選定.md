# デザインパターン選定

## 1. 対象
Windows デスクトップ GUI アプリとして、今回の開発に適したデザインパターンを比較し、採用方針を定義する。

## 2. 比較対象
- MVC
- MVP
- MVVM

## 3. 評価観点
- WinUI 3 / XAML との相性
- 3 ペイン画面とタイムライン同期の表現しやすさ
- 非モーダルサブ画面の状態管理
- 非同期処理と進捗表示の扱いやすさ
- テスト容易性
- OCR ワーカーや出力層との責務分離

## 4. 各パターンの評価
### 4.1 MVC
利点:
- 構造が理解しやすい
- 小規模アプリでは導入しやすい

懸念:
- WinUI 3 のデータバインディング活用が薄くなりやすい
- 画面状態や選択同期が Controller へ集まりやすい
- タイムライン、一覧、プレビューの相互同期で制御が肥大化しやすい

### 4.2 MVP
利点:
- View とロジックの分離がしやすい
- Presenter 単体のテストが比較的しやすい

懸念:
- WinUI 3 標準のバインディング機構を活かしにくい
- 画面イベントが増えるほど Presenter が肥大化しやすい
- タイムラインや非モーダル画面など複数の UI 状態をまたぐと仲介量が増えやすい

### 4.3 MVVM
利点:
- WinUI 3 / XAML のデータバインディングと相性が良い
- タイムライン、一覧、プレビューの選択状態を ViewModel で一元管理しやすい
- Command と状態プロパティで非同期実行や進捗表示を整理しやすい
- 非モーダル画面ごとに ViewModel を分けやすい
- GUI と OCR ワーカー、出力層をサービス経由で分離しやすい

懸念:
- ViewModel が肥大化しないよう責務分割が必要
- 単純な画面でもパターン導入コストは多少ある

## 5. 採用方針
今回のアプリケーションでは **MVVM** を採用する。

## 6. 採用理由
- WinUI 3 を前提としており、XAML バインディングを自然に使える。
- タイムライン、結果一覧、プレビュー、状態表示の同期を ViewModel で扱いやすい。
- OCR ワーカー呼び出しや出力処理をサービス層へ分離しやすい。
- 非同期実行、進捗表示、エラー表示を Command と状態プロパティで整理しやすい。
- UI テスト以前に、ViewModel 単位で状態遷移を検証しやすい。

## 7. 適用方針
- View
  - WinUI 3 / XAML で UI を構成する
- ViewModel
  - 画面状態、選択状態、Command を保持する
- Service
  - 動画入力、フレーム抽出、OCR、属性解析、統合、出力を担当する
- Worker Adapter
  - OCR ワーカーとの I/O 契約を吸収する

## 8. 実装上の注意
- 1 つの ViewModel に処理責務を詰め込みすぎない。
- タイムライン表示用の状態は専用 ViewModel へ分離する。
- OCR や出力処理は ViewModel から直接実装せず、サービス経由にする。
- GUI 表示専用モデルと永続化モデルを混在させない。
