# 処理パイプライン詳細設計

## 1. 対象
OCR ワーカー連携、処理順序、I/O 契約、エラー返却、再実行単位を定義する。

## 2. 処理パイプライン
1. 動画入力検証
2. 動画メタデータ取得
3. フレーム抽出
4. OCR ワーカーによる文字領域検出
5. OCR ワーカーによる文字認識
6. 属性解析
7. セグメント統合
8. JSON / CSV 出力
9. GUI 表示更新

初期実装では、OCR / 属性解析後に `TelopSegmentMerger` が同一文字列・同一属性キーの連続フレームをセグメントへ統合し、`ExportPackageWriter` がランディレクトリ配下の `output/` へ JSON / CSV を保存する。

## 3. OCR ワーカー連携方針
### 3.1 基本方針
- GUI 本体は OCR エンジンの詳細実装を直接呼び出さない。
- アプリケーション制御層が OCR ワーカーを呼び出し、共通契約へ正規化して GUI と後続処理へ渡す。
- OCR ワーカーは将来的な差し替えを前提に、入出力を共通形式にする。

### 3.2 呼び出し単位
- 初期実装ではフレーム単位呼び出しを基本とする。
- 将来、複数フレームのバッチ呼び出しへ拡張できる構成にする。

### 3.3 ワーカー責務
- 入力画像読込
- 文字領域検出
- 文字認識
- 検出結果の正規化
- 信頼度の付与
- エラー内容の返却

### 3.4 初期実装の接続方式
- アプリ本体は `IOcrWorkerClient` を通じて OCR ワーカーへ接続する。
- 外部ワーカー実行ファイルは `MOVIE_TELOP_OCR_WORKER` 環境変数で指定する。
- 外部ワーカーには `request.json` と `response.json` のパスを引数として渡す。
- `MOVIE_TELOP_OCR_WORKER` が未設定の場合は、抽出フレームと同名の `.ocr.json` サイドカーを同じ契約で読み込む。
- サイドカーも存在しない場合は、空検出の `success` 応答として扱い、GUI と後続処理の動作確認を継続できるようにする。
- PaddleOCR / Tesseract / Windows 標準 OCR の最終的な同梱方式は、テスト工程の精度検証と配布構成確認で再評価する。

## 5. ステータスと戻り値
### 5.1 共通ステータス
- `success`
- `warning`
- `error`

### 5.2 エラー返却形式
```json
{
  "code": "OCR_PROCESS_FAILED",
  "message": "OCR worker failed to process frame.",
  "details": "engine timeout",
  "recoverable": true
}
```

### 5.3 代表的なエラーコード
- `VIDEO_NOT_FOUND`
- `VIDEO_DECODE_FAILED`
- `FRAME_EXTRACT_FAILED`
- `OCR_WORKER_NOT_FOUND`
- `OCR_WORKER_START_FAILED`
- `OCR_RESPONSE_NOT_FOUND`
- `OCR_RESPONSE_READ_FAILED`
- `OCR_PROCESS_FAILED`
- `ATTRIBUTE_ANALYSIS_FAILED`
- `SEGMENT_MERGE_FAILED`
- `EXPORT_FAILED`

## 6. 再実行単位
1. 動画読込から全実行
2. フレーム抽出から再実行
3. OCR ワーカーから再実行
4. 属性解析から再実行
5. セグメント統合から再実行
6. 出力のみ再実行

再実行条件:
- 前段成果物が存在すること
- 入力設定の整合が取れていること
- ログ上で対象実行単位が追跡可能であること

初期実装では、GUI から以下を実行できる。
- `Rerun frames`: フレーム抽出から OCR、属性解析、セグメント統合、出力、ログ生成まで再実行する。
- `Rerun OCR`: 直近のフレーム抽出結果を使い、OCR、属性解析、セグメント統合、出力、ログ生成を再実行する。
- `Export only`: 直近の OCR / 属性解析 / セグメント結果を使い、出力とログ生成だけを再実行する。

入力検証、前段成果物不足、処理例外は GUI の `Last failure` に段階名、エラーコード、メッセージ、直近ログパスとして表示する。

## 7. ログ設計
### 7.1 ログに残す内容
- 実行開始時刻
- 実行終了時刻
- 動画ファイル名
- フレーム抽出設定
- OCR エンジン名
- 処理件数
- 警告件数
- エラー件数

### 7.3 初期実装の中間成果物
- フレーム画像: `work/runs/<run_id>/frames/`
- OCR リクエストとレスポンス: `work/runs/<run_id>/ocr/`
- 属性解析結果: `work/runs/<run_id>/attributes/`
- 実行ログ: `work/runs/<run_id>/logs/`
- 外部ワーカー未設定時も、同じ配置規約でリクエストとレスポンスを保存する。

### 7.4 初期実装の最終成果物
- JSON 出力: `work/runs/<run_id>/output/segments.json`
- セグメント CSV: `work/runs/<run_id>/output/segments.csv`
- フレーム CSV: `work/runs/<run_id>/output/frames.csv`
- 出力ファイルは OCR 検出がない場合も生成する。

### 7.5 初期実装のログ成果物
- 実行ログ: `work/runs/<run_id>/logs/run.log`
- 実行要約: `work/runs/<run_id>/logs/summary.json`
- ログには `run_id`、開始/終了時刻、入力動画、抽出間隔、OCR エンジン、フレーム数、検出数、セグメント数、警告数、エラー数、成果物パスを記録する。

### 7.2 ログ粒度
- 実行単位ログ
- フレーム単位ログ
- ワーカー呼び出し単位ログ

## 8. GUI 連携ポイント
- 処理開始時に進捗状態を `待機中` から `実行中` へ変更する。
- フレーム抽出完了件数を進捗表示へ反映する。
- OCR 失敗時は、対象フレームとエラー理由を GUI へ通知する。
- セグメント確定後は、結果一覧とタイムライン表示モデルを同時更新する。
