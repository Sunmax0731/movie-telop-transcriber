# テスト計画書

## 1. 文書情報
- システム名: movie-telop-transcriber
- 文書種別: テスト計画書
- 作成日: 2026-04-28
- 対象工程: テスト
- 対象 Issue: `#42`

## 2. 目的
実装工程で作成した GUI、動画入力、フレーム抽出、OCR ワーカー接続、属性解析、セグメント統合、出力、ログ、再実行制御が、要件と仕様に沿って動作するかを確認する。

## 3. テスト範囲
対象:
- WinUI 3 アプリの起動と主要画面表示
- 動画ファイル選択とメタデータ表示
- フレーム抽出と `work/runs/<run_id>/frames/` への保存
- OCR ワーカー I/O 契約と `.ocr.json` サイドカー読み込み
- 属性解析結果の内部モデル格納
- セグメント統合とタイムライン/一覧表示
- `segments.json`、`segments.csv`、`frames.csv` の出力
- `logs/run.log`、`logs/summary.json` の出力
- 設定ボタンから開く、設定と出力を統合した非モーダル子ウィンドウ表示
- フレーム抽出、OCR、出力の段階別再実行
- エラー表示と直近ログパス表示
- オフライン導入を想定した起動と実行

対象外:
- 実動画に対する OCR 精度の最終評価
- PaddleOCR / Tesseract / Windows 標準 OCR の最終採用判断
- 配布物作成と GitHub Release

## 4. 前提条件
- `dotnet build src/MovieTelopTranscriber.sln -p:Platform=x64` が成功すること。
- テスト工程用のサンプル動画と正解データは `#43` で整備する。
- 実 OCR エンジンが未導入の場合は、`.ocr.json` サイドカーで OCR レスポンスを代替する。
- オフライン導入検証は `#46` で、配布構成の確認と合わせて実施する。

## 5. テスト環境
- OS: Windows
- UI: WinUI 3 デスクトップアプリ
- Target Framework: `net10.0-windows10.0.26100.0`
- Platform: `x64`
- 主要依存:
  - Microsoft Windows App SDK
  - CommunityToolkit.Mvvm
  - OpenCvSharp4.Windows

## 6. テスト種別
### 6.1 ビルド確認
- ソリューションが復元、ビルドできること。
- 警告とエラーを記録すること。

判定基準:
- Debug x64 ビルドが成功する。
- 既知制約がない限り、エラーは 0 件であること。

### 6.2 機能テスト
- 動画選択後にファイル名、長さ、解像度、FPS、Codec が表示されること。
- 抽出間隔を変更してフレーム抽出数に反映されること。
- ランごとに `work/runs/<run_id>/` が作成されること。
- フレーム、OCR、属性、出力、ログが所定のディレクトリへ保存されること。

判定基準:
- 各成果物のパスが GUI とファイルシステムで一致する。
- ランが分離され、前回実行の成果物を上書きしない。

### 6.3 GUI 操作テスト
- メイン画面で動画選択、解析開始、タイムライン選択、一覧確認ができること。
- 設定ボタンから、設定と出力を統合した子ウィンドウを開けること。
- サブ画面表示中もメイン画面を操作できること。
- 設定と出力画面の抽出間隔変更がメイン画面へ反映されること。
- 設定と出力画面に JSON / CSV / ログパスと最新結果一覧が表示されること。

判定基準:
- 操作不能、クラッシュ、明らかな表示不整合がないこと。

### 6.4 OCR / 属性解析テスト
- `.ocr.json` サイドカーを配置したフレームで OCR レスポンスが読み込まれること。
- テキスト、信頼度、矩形情報が内部モデルへ格納されること。
- 矩形情報がある場合、フォントサイズ相当値が算出されること。
- 不明な色属性やフォント名は `null` として扱われること。

判定基準:
- サイドカーの入力値が JSON 出力と GUI 表示へ追跡できる。
- 不明値が空文字や欠落ではなく、仕様どおりに扱われる。

### 6.5 セグメント統合テスト
- 同一文字列・同一属性キーの連続フレームが 1 セグメントに統合されること。
- セグメントの開始時刻と終了時刻がフレーム時刻と抽出間隔から算出されること。
- 同一時刻に複数検出がある場合も一覧に表示されること。

判定基準:
- `segments.json` と GUI タイムラインの件数、時刻、文字列が一致する。

### 6.6 出力テスト
- `segments.json` が `schema_version`、`source_video`、`processing_settings`、`frames`、`segments`、`run_metadata` を含むこと。
- `segments.csv` と `frames.csv` が UTF-8 でヘッダー付き出力されること。
- OCR 検出が 0 件の場合も空配列とヘッダー付き CSV が生成されること。

判定基準:
- JSON がパース可能であること。
- CSV の列順が仕様と一致すること。

### 6.7 ログテスト
- `logs/run.log` と `logs/summary.json` が生成されること。
- `run_id`、入力動画、抽出間隔、OCR エンジン、件数、成果物パスが記録されること。
- GUI からログパスを確認できること。

判定基準:
- ログと実際の成果物パスが一致すること。

### 6.8 エラーと再実行テスト
- 無効な動画パスで入力検証エラーが表示されること。
- 前段成果物なしで `Rerun OCR` または `Export only` を押した場合、前段不足が表示されること。
- `Rerun frames` はフレーム抽出から後段まで再実行すること。
- `Rerun OCR` は直近フレームを使って OCR 以降を再実行すること。
- `Export only` は直近解析結果を使って出力とログ生成だけを再実行すること。

判定基準:
- `Last failure` に失敗段階、エラーコード、エラーメッセージが表示される。
- 再実行後の成果物が更新される。

### 6.9 オフライン導入テスト
- ネットワーク接続なしでも起動できること。
- 必要ランタイムと同梱物が整理されていること。
- サンプル動画で解析から出力まで実行できること。

判定基準:
- 導入手順に不足がないこと。
- 起動と主要機能がオフライン前提で成立すること。

## 7. テストデータ

## 8. 記録方法
- テスト結果は `docs/test-results/` 配下へ記録する。
- 手動 GUI テストは、実施日、実施者、環境、手順、期待結果、実結果、判定、補足を残す。
- コマンド実行結果は、実行コマンドと要約を記録する。
- 不具合または制約は `#47` で整理し、必要に応じて追加 Issue 化する。
