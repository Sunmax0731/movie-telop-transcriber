# 仕様書

## 1. 文書情報
- システム名: movie-telop-transcriber
- 文書種別: 仕様書
- 作成日: 2026-04-28
- 対象工程: 仕様検討

## 2. 仕様方針
- Windows 上で動作する GUI アプリとする。
- 主画面 1 画面で主要操作を完結させる。
- 結果確認、設定、詳細表示はサブ画面で提供する。
- サブ画面はメイン画面をブロックしない非モーダルダイアログ相当の扱いとする。
- 主要機能はオフライン環境で動作する。

## 3. 画面仕様
### 3.1 採用方針
- Windows 向けデザインシステムは Fluent 2 系を前提とする。
- 実装フレームワーク候補は WinUI 3 を第一候補とする。
- 標準コントロール、タイトルバー、ダイアログ、配色、アクセント挙動は Windows の標準体験に寄せる。
- メイン画面は WinUI 3 の標準レイアウトコンポーネントを中心に構成する。
- 独自装飾を優先せず、可読性と密度を重視する。

### 3.2 画面構成
メイン画面 1 画面で以下を縦または左右分割で配置する。
- 動画入力エリア
- 解析設定エリア
- 処理実行エリア
- 進捗表示エリア
- 抽出結果一覧エリア
- プレビューエリア

### 3.3 メイン画面レイアウト案
初期レイアウトは 3 ペイン構成を基本とする。
- 左ペイン
  - 動画入力
  - 解析設定
  - 実行操作
- 中央ペイン
  - 動画プレビュー
  - 現在フレーム情報
- 右ペイン
  - 抽出結果一覧
  - 選択中テロップ詳細

補足:
- 横幅不足時は、左ペインと右ペインを折りたたみ可能にする。
- 主要操作ボタンは左上から左中段に集約し、視線移動を短くする。

### 3.4 サブ画面方針
- 設定詳細
- 出力結果詳細
- エラー詳細
- 属性詳細

これらは非モーダルで開き、表示中もメイン画面を操作可能とする。

サブ画面の振る舞い:
- 同時に複数開ける構成を許容する。
- メイン画面の操作をブロックしない。
- 直前の表示位置とサイズを保存できる余地を持たせる。

### 3.5 メイン画面の主な操作
- 動画ファイル選択
- 抽出間隔設定
- 解析開始
- 解析停止
- 結果保存
- 結果一覧の選択
- 選択結果に対応する時刻プレビュー

### 3.6 メイン画面の主要表示項目
- 入力動画の基本情報
  - ファイル名
  - 長さ
  - 解像度
  - フレームレート
- 現在の処理状態
  - 待機中
  - 実行中
  - 完了
  - エラー
- 現在選択中の結果情報
  - 時刻
  - テキスト
  - 種別
  - 色属性
  - 信頼度

### 3.7 画面操作の基本原則
- 主要操作はメイン画面上で完結させる。
- 頻繁に使う設定はメイン画面に常設する。
- 詳細設定や補助情報のみをサブ画面へ逃がす。
- 実行中も結果一覧の参照や既存結果確認ができる構成を優先する。

### 3.8 WinUI 3 / Fluent 2 採用時の UI 方針
- 入力欄、設定欄、一覧、ボタン、通知は WinUI 3 標準コントロールを優先する。
- 情報のまとまりはカード乱用ではなく、グループ化と余白で整理する。
- 状態表示には `InfoBar` 相当の通知 UI を優先する。
- 一覧表示にはソートや列拡張余地のあるテーブル系 UI を想定する。
- テーマは Windows のライト / ダーク切替に追従できる構成を優先する。

### 3.9 設計工程への引き継ぎ事項
- ペインの最小幅と折りたたみ挙動
- プレビュー領域の再生操作有無
- 結果一覧の列定義
- サブ画面の永続化対象

## 4. 動画入力仕様
### 4.1 対応形式
第一対象は MP4 とする。加えて、一般的な動画拡張子を対応候補とする。
- `.mp4`
- `.mov`
- `.avi`
- `.mkv`
- `.wmv`
- `.m4v`

実際の対応可否は、採用するデコーダやライブラリの検証結果で確定する。

初期方針:
- 標準利用では `.mp4` を最優先とする。
- 他形式は「対応候補」であり、読み込み実績ベースで段階的に保証範囲を広げる。
- コンテナ拡張子だけでなく、実デコード成功をもって対応可否を判断する。

### 4.2 入力時の保持情報
- ファイルパス
- ファイル名
- コンテナ形式
- 再生時間
- 解像度
- フレームレート

追加保持項目:
- ファイルサイズ
- 映像コーデック情報
- 音声有無
- 最終更新日時

### 4.3 入力時のバリデーション
動画読み込み時は少なくとも以下を確認する。
- ファイルが存在すること
- 対応候補拡張子であること
- 動画ストリームを取得できること
- 再生時間、解像度、フレームレートを取得できること

読み込み失敗時は、少なくとも以下の理由を区別して表示する。
- ファイル未存在
- 非対応形式またはデコード失敗
- 動画ストリーム取得失敗
- メタデータ取得失敗

### 4.4 入力制約
- 初期仕様ではローカルファイル入力のみを対象とする。
- URL 入力やストリーミング入力は対象外とする。
- 複数動画の一括投入は初期仕様では対象外とする。

## 5. フレーム抽出仕様
### 5.1 基本方針
- デフォルトは 1 秒あたり 1 フレームとする。
- 利用者が設定で抽出間隔を変更できるようにする。
- 抽出は動画の先頭から終了までを基本とする。
- 開始時刻と終了時刻を指定した部分抽出にも対応できる構成にする。

### 5.2 設定項目案
- 抽出間隔秒数
- 開始時刻
- 終了時刻
- 出力先ディレクトリ
- 上書き可否

初期値:
- 抽出間隔秒数: `1.0`
- 開始時刻: `0`
- 終了時刻: 動画終端
- 出力先ディレクトリ: アプリ既定の作業ディレクトリ配下
- 上書き可否: `false`

将来拡張候補:
- キーフレーム基準抽出
- シーン変化検知抽出
- テロップ領域の変化量に応じた抽出補助

### 5.3 抽出結果
フレームごとに以下を保持する。
- `frame_index`
- `timestamp_ms`
- `image_path`

必要に応じて以下も保持対象とする。
- `source_width`
- `source_height`
- `extract_reason`

### 5.4 抽出ルール
- 時刻基準で抽出する。
- `1.0` 秒設定時は、`0ms`, `1000ms`, `2000ms` のように等間隔で抽出する。
- 動画終端が抽出間隔に一致しない場合、終端側の端数フレームは無理に追加しない。
- 同一時刻から再抽出した場合、同じ命名規則で同じ出力先に保存されることを前提とする。

### 5.5 フレームファイル命名規則
- 命名例: `frame_000123_004000ms.png`
- 構成要素:
  - 連番フレーム番号
  - 抽出時刻
  - 固定拡張子 `.png`

### 5.6 実装時の留意点
- フレーム番号基準ではなく時刻基準で管理し、可変フレームレート動画でも扱いやすくする。
- OCR 比較や再解析を考慮し、抽出設定をログへ残す。
- 抽出結果が大量になるため、出力先とクリーンアップ方針を設計工程で明確化する。

## 6. OCR 仕様方針
### 6.1 推奨案
第一推奨は、GUI 本体を Windows ネイティブ系で構築し、OCR はローカル実行の別コンポーネントとして分離する構成とする。

推奨順位:
1. PaddleOCR 系をローカル実行する構成
2. Tesseract OCR をローカル実行する構成
3. Windows 標準 OCR の適用可否を限定検証する構成

### 6.2 推奨理由
#### PaddleOCR 系
- 日本語を含む文字認識の柔軟性が比較的高い。
- テキスト検出と認識をまとめて扱いやすい。
- 完全オフライン構成を取りやすい。

懸念:
- Python 実行環境または推論環境の同梱設計が必要。
- 配布物が大きくなりやすい。

#### Tesseract OCR
- オフライン導入が容易で実績が多い。
- 配布構成が比較的単純。

懸念:
- 装飾の強いテロップや縁取り文字で精度が落ちやすい。
- 検出より認識寄りで、領域切り出し側の工夫が必要。

#### Windows 標準 OCR
- Windows 標準 API に寄せやすい。

懸念:
- 対応言語や認識品質、装飾文字対応の点で制約が残る可能性が高い。

### 6.3 現時点の提案
- 初期仕様は `WinUI 3 + ローカル OCR ワーカー` を推奨する。
- OCR の第一候補は PaddleOCR 系とし、比較対象として Tesseract OCR を評価する。
- 実装前に少量の実動画サンプルで比較検証を行う。

### 6.4 比較観点
候補比較では以下の観点を必須とする。
- 認識精度
  - 日本語テロップの完全一致率
  - 英数字混在時の崩れにくさ
- 検出性能
  - 文字領域検出の安定性
  - 複数行テロップへの追従性
- 装飾耐性
  - 縁取り文字
  - 地付き文字
  - 低コントラスト背景
- 導入性
  - オフライン配布の容易さ
  - 依存ランタイムの重さ
  - 初回セットアップの複雑さ
- 運用性
  - エラー時の切り分けやすさ
  - 将来の差し替えやすさ
- 性能
  - 処理速度
  - メモリ消費傾向
  - CPU 使用率傾向

### 6.5 採用判断
初期採用方針は以下とする。
- 第一候補: PaddleOCR 系
- 第二候補: Tesseract OCR
- 参考比較: Windows 標準 OCR

判断理由:
- テロップ用途では、認識だけでなく文字領域検出も重要である。
- 日本語と装飾文字への耐性を優先すると、初期比較では PaddleOCR 系が最も本命になりやすい。
- Tesseract OCR は配布と導入が比較的簡単なため、軽量構成の代替候補として価値がある。
- Windows 標準 OCR は Windows 統合の魅力はあるが、本システムの主対象である装飾付き日本語テロップへの適合性が不確実である。

### 6.6 実装方針
- GUI 本体は OCR エンジン実装に直接依存しすぎない構成にする。
- OCR はワーカー層として抽象化し、将来的にエンジンを差し替え可能にする。
- 初期実装では PaddleOCR 系を前提に I/O 契約を定義する。
- 比較検証の結果、導入性または精度で問題が大きい場合は Tesseract OCR へ切り替え可能とする。

### 6.7 実装前に確認する項目
- Windows 配布時に必要な OCR 依存物の同梱方法
- オフライン端末での初回起動時間
- 日本語モデルのサイズ
- ログ出力と失敗時の再実行導線

## 7. テロップ属性保持形式
### 7.1 基本方針
- フレーム単位結果と、統合済みテロップ単位結果を分けて保持する。
- 色はすべて `#RRGGBB` 形式で保持する。
- 不明値は `null` を許容する。
- 属性名は JSON / CSV / 内部モデルで可能な限り共通化する。
- 推定値と確定値を混同しないため、推定項目であることを前提に扱う。

### 7.2 フレーム単位属性
- `text`
- `timestamp_ms`
- `frame_index`
- `font_family`
- `font_size`
- `text_color`
- `stroke_color`
- `background_color`
- `confidence`

### 7.3 統合テロップ単位属性
- `segment_id`
- `start_timestamp_ms`
- `end_timestamp_ms`
- `text`
- `text_type`
- `font_family`
- `font_size`
- `text_color`
- `stroke_color`
- `background_color`
- `confidence`

### 7.4 属性の型定義
#### 共通属性
- `text`
  - 型: `string`
  - 内容: OCR により取得した文字列
- `confidence`
  - 型: `number`
  - 範囲: `0.0` から `1.0`
  - 不明時: `null`

#### 時刻・識別系
- `frame_index`
  - 型: `integer`
  - 内容: 抽出対象フレームの連番
- `timestamp_ms`
  - 型: `integer`
  - 内容: フレーム抽出時刻のミリ秒
- `segment_id`
  - 型: `string`
  - 例: `seg-0001`
- `start_timestamp_ms`
  - 型: `integer`
- `end_timestamp_ms`
  - 型: `integer`

#### フォント系
- `font_family`
  - 型: `string | null`
  - 内容: 推定フォント名または分類名
- `font_size`
  - 型: `number | null`
  - 初期方針: 推定ピクセル高さに近い値で保持する
- `font_size_unit`
  - 型: `string | null`
  - 初期値候補: `px`

#### 色系
- `text_color`
  - 型: `string | null`
- `stroke_color`
  - 型: `string | null`
- `background_color`
  - 型: `string | null`

色系属性の値規約:
- 形式は `#RRGGBB` とする。
- 英字は大文字で正規化する。
- アルファ値は初期仕様では保持しない。
- 複数色グラデーションは初期仕様では代表色 1 値に丸める。

#### 種別系
- `text_type`
  - 型: `string | null`
  - 初期候補値:
    - `standard`
    - `title`
    - `name`
    - `caption_band`
    - `unknown`

### 7.5 不明値の扱い
- 値を取得できなかった場合は空文字ではなく `null` を使用する。
- OCR 結果が空で属性のみ残る状態は初期仕様では許容しない。
- `text_type` を判定できない場合は `null` ではなく `unknown` を許容する。

### 7.6 フレーム単位とセグメント単位の関係
- フレーム単位属性は抽出時点の観測値とする。
- セグメント単位属性は複数フレームを統合した代表値とする。
- セグメント単位の色、フォントサイズ、信頼度は、統合対象フレームから代表値を算出する。
- 代表値算出方式の詳細は設計工程で定めるが、初期仕様では最頻値または中央値を優先候補とする。

### 7.7 補足
- `font_family` は厳密一致ではなく推定値とする。
- `font_size` は初期仕様ではピクセル相当値を採る。
- `text_type` は将来的に「通常テロップ」「人物名」「見出し」などの分類へ拡張可能とする。

## 8. JSON 出力スキーマ案
### 8.1 基本方針
- JSON は機械可読な完全出力とする。
- フレーム単位情報とセグメント単位情報を同一ファイルで保持できる構成とする。
- 将来拡張のため、トップレベルに `schema_version` を必須とする。
- 不明値は `null` を使用し、未定義項目の省略は原則行わない。

### 8.2 トップレベル構造
- `schema_version`
- `source_video`
- `processing_settings`
- `frames`
- `segments`
- `run_metadata`

### 8.3 トップレベル項目定義
- `schema_version`
  - 型: `string`
  - 例: `1.0.0`
- `source_video`
  - 型: `object`
  - 内容: 入力動画のメタデータ
- `processing_settings`
  - 型: `object`
  - 内容: 抽出設定、OCR 設定、実行条件
- `frames`
  - 型: `array`
  - 内容: フレーム単位解析結果
- `segments`
  - 型: `array`
  - 内容: 統合済みテロップ単位解析結果
- `run_metadata`
  - 型: `object`
  - 内容: 実行ログ要約、生成日時、アプリバージョンなど

### 8.4 配列要素の扱い
- `frames` は抽出時刻順に並べる。
- `segments` は `start_timestamp_ms` 昇順に並べる。
- 空データの場合も配列自体は出力する。

```json
{
  "schema_version": "1.0.0",
  "source_video": {
    "file_path": "D:/videos/sample.mp4",
    "file_name": "sample.mp4",

    "width": 1920,
    "height": 1080,
    "fps": 29.97
  },
  "processing_settings": {
    "frame_interval_seconds": 1.0,
    "ocr_engine": "paddleocr",
    "offline_mode": true
  },
  "frames": [
    {
      "frame_index": 30,
      "timestamp_ms": 1000,
      "image_path": "work/frames/frame_000030.png",
      "detections": [
        {
          "text": "サンプルテロップ",
          "font_family": "Yu Gothic UI",
          "font_size": 42,
          "text_color": "#FFFFFF",
          "stroke_color": "#000000",
          "background_color": "#FFCC00",
          "confidence": 0.94
        }
      ]
    }
  ],
  "segments": [
    {
      "segment_id": "seg-0001",
      "start_timestamp_ms": 1000,
      "end_timestamp_ms": 4000,
      "text": "サンプルテロップ",
      "text_type": "standard",
      "font_family": "Yu Gothic UI",
      "font_size": 42,
      "text_color": "#FFFFFF",
      "stroke_color": "#000000",
      "background_color": "#FFCC00",

    }
  ],
  "run_metadata": {
    "generated_at": "2026-04-28T07:00:00+09:00",
    "application_version": "0.1.0",
    "processing_time_ms": 18250
  }
}
```

### 8.5 `run_metadata` 項目案
- `generated_at`
  - 型: `string`
  - 形式: ISO 8601
- `application_version`
  - 型: `string`
- `processing_time_ms`
  - 型: `integer | null`
- `warning_count`
  - 型: `integer | null`
- `error_count`
  - 型: `integer | null`

### 8.6 スキーマバージョン方針
- 初期バージョンは `1.0.0` とする。
- 後方互換を壊さない項目追加はマイナー更新とする。
- 既存項目の意味変更や削除はメジャー更新とする。

## 9. CSV 出力スキーマ案
### 9.1 基本方針
- CSV は人間が表計算ソフトで確認しやすい派生出力とする。
- 標準出力はセグメント単位 CSV とする。
- 必要に応じてフレーム単位 CSV を別ファイルで出力する。
- 1 ファイルに異なる粒度のデータを混在させない。

### 9.2 セグメント出力
標準の CSV 出力は、統合済みテロップ単位を 1 行とする。

列案:
- `segment_id`
- `start_timestamp_ms`
- `end_timestamp_ms`
- `text`
- `text_type`
- `font_family`
- `font_size`
- `text_color`
- `stroke_color`
- `background_color`
- `confidence`

列順方針:
- 時系列確認しやすいよう、識別子、時刻、文字列、属性、信頼度の順に固定する。

### 9.3 フレーム単位出力
必要に応じて別 CSV として出力する。

列案:
- `frame_index`
- `timestamp_ms`
- `text`
- `font_family`
- `font_size`
- `text_color`
- `stroke_color`
- `background_color`
- `confidence`

### 9.4 CSV ファイル名案
- セグメント単位: `segments.csv`
- フレーム単位: `frames.csv`

### 9.5 CSV 出力ルール
- 文字コードは UTF-8 とする。
- 1 行目はヘッダー行とする。
- 不明値は空文字ではなく空欄出力とする。
- 色は JSON と同じく `#RRGGBB` 形式で出力する。
- 改行を含む OCR 結果は、CSV 上では改行をスペースに正規化してもよい。

### 9.6 JSON と CSV の役割分担
- JSON を完全データとして扱う。
- CSV は確認・集計用の簡易出力とする。
- JSON にある補助情報すべてを CSV に載せることは必須としない。

## 11. OCR 比較検証用サンプル動画条件
### 11.1 目的
PaddleOCR 系、Tesseract OCR、必要に応じて Windows 標準 OCR の比較を行うため、最低限そろえるサンプル動画条件を定義する。

### 11.2 検証対象のテロップ種類
少なくとも以下の種類を含める。
- 通常テロップ
  - 単色文字
  - 枠付き文字
- 見出し系テロップ
  - 大きい文字
  - 強い装飾を持つ文字
- 人名や地名などの短文テロップ
- 複数行テロップ
- 画面下帯付きテロップ
- 背景映像とのコントラストが低いテロップ

### 11.3 検証対象動画の条件
少なくとも以下の条件を含める。
- 解像度
  - 1920x1080
  - 1280x720
- フレームレート
  - 29.97fps 付近
  - 30fps
- 文字言語
  - 日本語
  - 日本語と英数字の混在
- 表示位置
  - 下部
  - 上部
  - 画面中央付近
- 背景条件
  - 背景が比較的静的
  - 背景が動いている
  - 明部背景
  - 暗部背景

### 11.4 最低限用意するサンプル本数
初期比較では最低 5 本を用意する。
- 1 本目: 標準的な下部テロップ中心
- 2 本目: 装飾や縁取りが強いテロップ中心
- 3 本目: 背景変化が大きい映像
- 4 本目: 複数行や帯付きテロップ中心
- 5 本目: 日本語と英数字が混在するテロップ中心

### 11.5 1 本あたりの長さ目安
- 30 秒から 120 秒を目安とする。
- 長すぎる動画は初期比較のコストを上げるため避ける。
- テロップの種類ごとに十分な出現回数があることを優先する。

### 11.6 評価観点
OCR エンジン比較では少なくとも以下を記録する。
- 文字認識精度
  - 完全一致率
  - 文字単位の誤認識傾向
- 検出安定性
  - テロップ未検出の有無
  - 同一テロップの重複検出傾向
- 装飾耐性
  - 縁取り文字
  - 地付き文字
  - 低コントラスト文字
- 処理性能
  - 処理時間
  - CPU 使用率の傾向
  - メモリ使用量の傾向
- 導入性
  - オフライン配布のしやすさ
  - 依存ライブラリの重さ

### 11.7 比較記録の単位
- 動画単位
- フレーム単位
- テロップセグメント単位

比較結果は少なくとも以下を保存対象とする。
- 使用 OCR エンジン名
- 入力動画名
- 抽出間隔
- 認識結果
- 正解データとの差分メモ
- 処理時間
- 判定メモ

### 11.8 正解データの扱い
- 比較用サンプルには手作業の正解データを用意する。
- 正解データには少なくとも文字列、開始時刻、終了時刻を含める。
- 色やフォントなどの属性比較は、初期段階では代表サンプルに限定してもよい。
