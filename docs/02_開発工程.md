# 開発工程

## 1. 方針
本システムは、一般的なシステム開発工程に沿って段階的に進める。各工程で成果物と確認観点を定義し、次工程へ進む条件を明確にする。

## 2. 工程一覧
1. 要件定義
2. 仕様検討
3. 設計
4. 実装
5. テスト
6. リリース

## 3. 各工程の内容
### 3.1 要件定義
目的:
- システムの目的、対象範囲、利用者、機能要件、非機能要件を定める。

主な成果物:
- `docs/01_要件定義.md`

完了条件:
- GUI 前提、オフライン前提、主要機能、制約条件が文書化されていること。
- 次工程である仕様検討の親 Issue と初期タスク Issue が起票されていること。
- 工程完了レビューとして `AGENTS.md`、`SKILL.md`、`README.md`、関連 `docs/` の整合確認が終わっていること。

### 3.2 仕様検討
目的:
- 要件を満たすための具体仕様を定める。

主な検討項目:
- 動画入力仕様
- フレーム抽出仕様
- OCR 処理仕様
- テロップ属性記録仕様
- 結果統合仕様
- 出力形式仕様
- エラー時の扱い

主な成果物:
- 仕様書
- データスキーマ案

完了条件:
- 実装者が画面、処理、データ形式を迷わず作業できる粒度になっていること。
- 次工程である設計の親 Issue と初期タスク Issue が起票されていること。
- 工程完了レビューとして `AGENTS.md`、`SKILL.md`、`README.md`、関連 `docs/` の整合確認が終わっていること。

### 3.3 設計
目的:
- 仕様を実装可能な構成に落とし込む。

主な検討項目:
- 画面設計
- モジュール設計
- データ構造設計
- ファイル配置設計
- ログ設計

主な成果物:
- 基本設計書
- 詳細設計書
- 画面遷移または画面構成案

完了条件:
- 実装タスクへ分割可能であり、依存関係が整理されていること。
- 次工程である実装の親 Issue と初期タスク Issue が起票されていること。
- 工程完了レビューとして `AGENTS.md`、`SKILL.md`、`README.md`、関連 `docs/` の整合確認が終わっていること。

### 3.4 実装
目的:
- 設計に基づきアプリケーションを構築する。

主な作業:
- GUI 実装
- 動画読み込み実装
- フレーム抽出実装
- OCR 実装
- 属性解析実装
- 統合処理実装
- 出力実装

完了条件:
- 仕様上の対象機能が動作し、既知制約が記録されていること。
- 次工程であるテストの親 Issue と初期タスク Issue が起票されていること。
- 工程完了レビューとして `AGENTS.md`、`SKILL.md`、`README.md`、関連 `docs/` の整合確認が終わっていること。

### 3.5 テスト
目的:
- 仕様と実装の整合を確認し、品質を担保する。

主なテスト:
- 単体テスト
- 結合テスト
- GUI 操作テスト
- オフライン動作確認

主な成果物:
- テスト計画書
- テスト結果
- 既知不具合一覧

完了条件:
- 主要シナリオが通過し、重大不具合が解消されていること。
- 次工程であるリリースの親 Issue と初期タスク Issue が起票されていること。
- 工程完了レビューとして `AGENTS.md`、`SKILL.md`、`README.md`、関連 `docs/` の整合確認が終わっていること。

### 3.6 リリース
目的:

主な作業:

完了条件:
- オフライン環境で導入と利用が可能な配布形態が準備されていること。
- 工程完了レビューとして `AGENTS.md`、`SKILL.md`、`README.md`、関連 `docs/` の整合確認が終わっていること。

## 5. Issue 駆動運用
- GitHub 上に工程ごとのマイルストーンを作成する。
- 各工程には親 Issue を 1 件作成し、工程全体の完了条件と成果物を管理する。
- 現在工程の具体タスクは Issue として起票する。
- Issue は、文書更新、比較検証、方針決定、実装準備などの完了単位で分割する。
- 原則として Issue ごとに専用ブランチを切って作業する。
- ブランチ名は `issue-<番号>-<要約>` 形式を基本とする。
- 工程完了時は、対応 Issue の完了と成果物の整合を確認してから次工程へ進む。
- 工程完了時は、次工程の親 Issue と初期タスク Issue を先に起票する。

## 6. 工程境界レビュー
- 各工程の完了時に、少なくとも以下を見直す。
  - `AGENTS.md`
  - `SKILL.md`
  - `README.md`
  - 当該工程に関連する `docs/`
- 見直しでは、現在の方針、GitHub 運用、実装状態、次工程の開始条件に齟齬がないことを確認する。
