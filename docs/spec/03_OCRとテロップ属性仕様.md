# OCRとテロップ属性仕様

## 1. OCR 仕様方針
### 1.1 推奨案
第一推奨は、GUI 本体を Windows ネイティブ系で構築し、OCR はローカル実行の別コンポーネントとして分離する構成とする。

推奨順位:
1. PaddleOCR 系をローカル実行する構成
2. Tesseract OCR をローカル実行する構成
3. Windows 標準 OCR の適用可否を限定検証する構成

### 1.2 推奨理由
#### PaddleOCR 系
- 日本語を含む文字認識の柔軟性が比較的高い。
- テキスト検出と認識をまとめて扱いやすい。
- 完全オフライン構成を取りやすい。

懸念:
- Python 実行環境または推論環境の同梱設計が必要。
- 配布物が大きくなりやすい。

#### Tesseract OCR
- オフライン導入が容易で実績が多い。
- 配布構成が比較的単純。

懸念:
- 装飾の強いテロップや縁取り文字で精度が落ちやすい。
- 検出より認識寄りで、領域切り出し側の工夫が必要。

#### Windows 標準 OCR
- Windows 標準 API に寄せやすい。

懸念:
- 対応言語や認識品質、装飾文字対応の点で制約が残る可能性が高い。

### 1.3 現時点の提案
- 初期仕様は `WinUI 3 + ローカル OCR ワーカー` を推奨する。
- OCR の第一候補は PaddleOCR 系とし、比較対象として Tesseract OCR を評価する。
- 実装前に少量の実動画サンプルで比較検証を行う。

### 1.4 比較観点
候補比較では以下の観点を必須とする。
- 認識精度
  - 日本語テロップの完全一致率
  - 英数字混在時の崩れにくさ
- 検出性能
  - 文字領域検出の安定性
  - 複数行テロップへの追従性
- 装飾耐性
  - 縁取り文字
  - 地付き文字
  - 低コントラスト背景
- 導入性
  - オフライン配布の容易さ
  - 依存ランタイムの重さ
  - 初回セットアップの複雑さ
- 運用性
  - エラー時の切り分けやすさ
  - 将来の差し替えやすさ
- 性能
  - 処理速度
  - メモリ消費傾向
  - CPU 使用率傾向

### 1.5 採用判断
初期採用方針は以下とする。
- 第一候補: PaddleOCR 系
- 第二候補: Tesseract OCR
- 参考比較: Windows 標準 OCR

判断理由:
- テロップ用途では、認識だけでなく文字領域検出も重要である。
- 日本語と装飾文字への耐性を優先すると、初期比較では PaddleOCR 系が最も本命になりやすい。
- Tesseract OCR は配布と導入が比較的簡単なため、軽量構成の代替候補として価値がある。
- Windows 標準 OCR は Windows 統合の魅力はあるが、本システムの主対象である装飾付き日本語テロップへの適合性が不確実である。

### 1.6 実装方針
- GUI 本体は OCR エンジン実装に直接依存しすぎない構成にする。
- OCR はワーカー層として抽象化し、将来的にエンジンを差し替え可能にする。
- 初期実装では PaddleOCR 系を前提に I/O 契約を定義する。
- 比較検証の結果、導入性または精度で問題が大きい場合は Tesseract OCR へ切り替え可能とする。

### 1.6.1 初期実装状態
- GUI 本体は `IOcrWorkerClient` 経由で OCR ワーカーへ接続する。
- `MOVIE_TELOP_OCR_ENGINE=paddleocr` の場合、または OCR エンジン未指定かつ外部 worker 未指定の場合は `PaddleOcrWorkerClient` を使う。
- 外部 OCR ワーカーは `MOVIE_TELOP_OCR_WORKER` 環境変数で差し替え可能とする。
- `json-sidecar` 検証を行う場合は `MOVIE_TELOP_OCR_ENGINE=json-sidecar` を明示し、抽出フレームと同名の `.ocr.json` サイドカーを読み込む。
- `json-sidecar` 検証時に sidecar がない場合は `OCR_SIDECAR_NOT_FOUND` の `error` 応答を作成し、OCR 未実行を空検出の正常完了として扱わない。
- 実 OCR エンジンの初期配布形態は #48 で確定済み。アプリ本体 zip には `tools/ocr/paddle_ocr_worker.py` を同梱し、Python runtime、PaddlePaddle / PaddleOCR の Python package、PaddleOCR モデル本体は同梱しない。

### 1.7 実装前に確認する項目
- Windows 配布時に必要な OCR 依存物の事前配置方法
- オフライン端末での初回起動時間
- 日本語モデルのサイズ
- ログ出力と失敗時の再実行導線

## 2. テロップ属性保持形式
### 2.1 基本方針
- フレーム単位結果と、統合済みテロップ単位結果を分けて保持する。
- 属性スキーマは将来拡張を見越して色/フォント/背景の項目を保持する。初期リリースの色系属性は厳密な `#RRGGBB` ではなく分類ラベルとして扱う。
- 不明値は `null` を許容する。
- 属性名は JSON / CSV / 内部モデルで可能な限り共通化する。
- 推定値と確定値を混同しないため、推定項目であることを前提に扱う。

### 2.2 フレーム単位属性
- `text`
- `timestamp_ms`
- `frame_index`
- `font_family`
- `font_size`
- `text_color`
- `stroke_color`
- `background_color`
- `confidence`

### 2.3 統合テロップ単位属性
- `segment_id`
- `start_timestamp_ms`
- `end_timestamp_ms`
- `text`
- `text_type`
- `font_family`
- `font_size`
- `text_color`
- `stroke_color`
- `background_color`
- `confidence`

### 2.4 属性の型定義
#### 共通属性
- `text`
  - 型: `string`
  - 内容: OCR により取得した文字列
- `confidence`
  - 型: `number`
  - 範囲: `0.0` から `1.0`
  - 不明時: `null`

#### 時刻・識別系
- `frame_index`
  - 型: `integer`
  - 内容: 抽出対象フレームの連番
- `timestamp_ms`
  - 型: `integer`
  - 内容: フレーム抽出時刻のミリ秒
- `segment_id`
  - 型: `string`
  - 例: `seg-0001`
- `start_timestamp_ms`
  - 型: `integer`
- `end_timestamp_ms`
  - 型: `integer`

#### フォント系
- `font_family`
  - 型: `string | null`
  - 内容: 推定フォント名または分類名
- `font_size`
  - 型: `number | null`
  - 初期方針: 推定ピクセル高さに近い値で保持する
- `font_size_unit`
  - 型: `string | null`
  - 初期値候補: `px`

#### 色系
- `text_color`
  - 型: `string | null`
  - 初期リリース: `白文字`、`黒文字`、`緑文字`、`黄文字`、`赤文字`、`青文字` などの分類ラベル
- `stroke_color`
  - 型: `string | null`
  - 初期リリース: `黒枠`、`白枠` などの分類ラベル
- `background_color`
  - 型: `string | null`
  - 初期リリース: `null`

色系属性の値規約:
- 初期リリースでは、OCR bounding box 内の色分布から色名/枠名の日本語ラベルへ分類する。
- 厳密な `#RRGGBB` 値、アルファ値、グラデーション、背景帯の代表色は初期リリースでは保証しない。
- 将来、正確な代表色抽出を実装する場合は、`#RRGGBB` 値へ拡張する。

#### 種別系
- `text_type`
  - 型: `string | null`
  - 初期候補値:
    - `未分類`
    - `白文字`
    - `黒文字`
    - `緑文字`
    - `黄文字`
    - `赤文字`
    - `青文字`
    - `黒枠`
    - `白枠`
    - `白文字 / 黒枠`
    - `黄文字 / 黒枠`

### 2.5 不明値の扱い
- 値を取得できなかった場合は空文字ではなく `null` を使用する。
- OCR 結果が空で属性のみ残る状態は初期仕様では許容しない。
- `text_type` を判定できない場合は `null` ではなく `未分類` を使用する。

### 2.6 フレーム単位とセグメント単位の関係
- フレーム単位属性は抽出時点の観測値とする。
- セグメント単位属性は複数フレームを統合した代表値とする。
- セグメント単位の色、フォントサイズ、信頼度は、統合対象フレームから代表値を算出する。
- 代表値算出方式の詳細は設計工程で定めるが、初期仕様では最頻値または中央値を優先候補とする。

初期実装では、OCR の矩形座標から推定できる場合に `font_size` をピクセル高さ相当として保持する。色属性は OCR bounding box 内の色分布から暫定的な色/枠ラベルとして推定し、フォント名と背景色は外部ワーカーまたは後続の画像解析実装が値を返すまで `null` を許容する。

初期リリースで保証する属性範囲と既知制約は `docs/spec/07_テロップ属性リリース範囲.md` を正とする。

### 2.7 補足
- `font_family` は厳密一致ではなく推定値とする。
- `font_size` は初期仕様ではピクセル相当値を採る。
- `text_type` は初期リリースでは色/枠ラベルであり、将来的に「通常テロップ」「人物名」「見出し」などの意味分類へ拡張可能とする。
