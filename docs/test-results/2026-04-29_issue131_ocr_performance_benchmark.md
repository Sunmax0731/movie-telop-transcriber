# Issue 131 OCR 処理速度ベンチマーク

## 概要
- 対象 Issue: `#131`
- 計測日: 2026-04-29
- 目的: OCR 処理の主要内訳を把握し、次に着手する高速化タスクを決める。

## 今回の対応
- `summary.json` と `run.log` に OCR 性能サマリを出力するようにした。
- `logs/ocr-performance.json` を追加し、フレーム単位で次の計測値を保存するようにした。
  - request write
  - worker initialization
  - worker execution
  - response read
  - attribute analysis
  - attribute write
  - total

- device: `cpu`
- language: `ja`
- preprocess: `true`
- contrast: `1.1`
- sharpen: `true`

  - 初回フレーム `32.5s`

  - 初回フレーム `3.9s`

  - 一部フレームで `12s` 前後のスパイクあり

## ボトルネック分析
1. 主律速は OCR worker 実行
   - どのケースでも `ocr_total_ms` のほぼ全てを `worker_execution_ms` が占める。
   - 属性解析、属性書出し、応答読込は相対的に小さい。
2. 初回 warmup コストが大きい
   - 短尺では初回 1 フレームが全体時間を大きく押し上げる。
   - 中尺でも先頭の `14.8s` が目立つ。
3. 定常時でも 1 フレームあたり約 2.7 から 3.0 秒
   - フレーム数をそのまま増やす構成では処理時間が線形に伸びる。
   - 間引きや候補選別なしでは長尺動画で待ち時間が重い。
4. 長尺では定常中にもスパイクがある
   - 一部フレームで `8s` から `13s` 台の遅延が発生した。
   - worker 内部の負荷変動、前処理条件、CPU 利用状況などの追加調査余地がある。

## 判断
- まず優先して削るべき対象は OCR worker 実行時間であり、属性解析や書出しの最適化優先度は低い。
- 改善効果を最も出しやすいのは初回 warmup の吸収。
- 中長尺では、OCR に渡すフレーム数そのものを減らす候補選別が有力。
- 並列化やバッチ化は有望だが、worker の安定性と CPU 使用量の検証が先に必要。
