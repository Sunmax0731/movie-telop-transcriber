# 配布方式と導入構成の再評価

## 1. 基本情報
- 対象 Issue: `#178`、`#179`
- 実施日: 2026-04-30
- 対象: 配布方式再評価、OCR runtime 導入構成再整理
- 記録者: Codex

## 2. 検証結果

### 2.1 self-contained publish 再検証
実行コマンド:

```powershell
dotnet publish src\MovieTelopTranscriber.App\MovieTelopTranscriber.App.csproj `
  -c Release `
  -p:Platform=x64 `
  -r win-x64 `
  --self-contained true `
  -o temp\self-contained-recheck
```

起動確認:

```powershell
$p = Start-Process `
  -FilePath "temp\self-contained-recheck\MovieTelopTranscriber.App.exe" `
  -WorkingDirectory "temp\self-contained-recheck" `
  -PassThru
Start-Sleep -Seconds 30
if ($p.HasExited) { $p.ExitCode }
```

結果:
- publish 自体は成功した。
- 出力ファイル数は `555`、合計サイズは `279,022,376 bytes` だった。
- 起動後 5 秒時点では残存していたが、30 秒以内に終了した。
- 終了コードは `-1073741189` で、16 進表記では `0xC000027B` だった。

判断:
- 2026-04-28 の旧検証と同じ系統の失敗が再現したため、`self-contained publish` は現時点でも採用しない。
- 5 秒時点の見かけ上の起動継続だけでは成立判定に使わない。

### 2.2 Release build + インストーラ経路の再確認
実行コマンド:

```powershell
powershell -NoProfile -ExecutionPolicy Bypass `
  -File .\tools\install\Install-MovieTelopTranscriber.ps1 `
  -PackageZipPath .\dist\movie-telop-transcriber-win-x64-v0.1.4.zip `
  -InstallRoot .\temp\issue178-install\MovieTelopTranscriber `
  -OcrRuntimeRoot .\temp\issue178-install\ocr-runtime `
  -SkipModelDownload `
  -Force
```

追加確認:

```powershell
temp\issue178-install\ocr-runtime\.venv\Scripts\python.exe `
  -c "import paddleocr, paddle; print('ok')"
```

```powershell
$p = Start-Process `
  -FilePath "temp\issue178-install\MovieTelopTranscriber\app\MovieTelopTranscriber.App.exe" `
  -WorkingDirectory "temp\issue178-install\MovieTelopTranscriber\app" `
  -PassThru
Start-Sleep -Seconds 5
if (-not $p.HasExited) { Stop-Process -Id $p.Id -Force }
```

結果:
- zip 展開、アプリ配置、Python 仮想環境作成、`paddlepaddle==3.2.0`、`paddleocr==3.5.0` の導入が成功した。
- `movie-telop-transcriber.settings.json` が導入先の `python.exe` と `paddle_ocr_worker.py` を指す内容で生成された。
- `import paddleocr, paddle` は成功した。
- 導入後アプリは 5 秒時点で起動継続していた。

判断:
- 現行の標準配布経路は `Release build を含む zip` と `Install-MovieTelopTranscriber.ps1/cmd` の組み合わせで維持する。
- OCR runtime はアプリ本体と分離したディレクトリへ置く現在構成を採用する。

## 3. 配布方式比較

| 方式 | 利点 | 欠点 | 今回の判断 |
| --- | --- | --- | --- |
| Release build zip + installer | 現行実装と整合し、起動と導入の実測結果がある | .NET Desktop Runtime 前提、配布 zip が大きい | 採用 |
| self-contained publish | .NET runtime 前提を減らせる可能性がある | 30 秒以内に `0xC000027B` で終了し、現行では不安定 | 不採用 |
| アプリ zip + OCR runtime 分離配布 | OCR runtime を再利用しやすく、アプリ更新を軽くできる | 利用者が 2 つの資材を扱う必要がある | フォールバック候補 |
| 完全オフライン一括パック | 制限端末へ持ち込みやすい | 配布サイズ、更新、検証負荷が大きい | 現時点では採用しない |

## 4. OCR runtime 導入構成比較

| 構成 | 想定利用者 | 利点 | 欠点 | 今回の判断 |
| --- | --- | --- | --- | --- |
| オンライン installer 標準導入 | 一般利用者 | 手順が最短で、app / runtime / settings を一括で揃えられる | ネットワーク接続と Python 3.10 が必要 | 標準採用 |
| 手動導入 / オフライン導入 | 管理端末、閉域端末 | 接続制限下でも運用できる | 手順が長く、設定ミス余地が大きい | フォールバック |
| 事前構築済み runtime pack | 管理者、再配布担当 | 再導入や横展開がしやすい | release asset と検証手順の整備が別途必要 | 将来候補 |

## 5. 結論
- 配布方式は、引き続き `Release build zip + installer` を標準とする。
- `self-contained publish` は再検証しても成立しなかったため、次回 release 採用対象にはしない。
- OCR runtime 導入構成は、`アプリ本体` と `ocr-runtime` を分離した現在構成を維持する。
- 標準導入はオンライン installer、手動導入とオフライン導入はフォールバックとして文書化する。
