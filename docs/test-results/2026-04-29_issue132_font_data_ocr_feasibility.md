# Issue 132 フォントデータ参照による OCR 精度向上可否の調査

## 対象
- 対象 Issue: `#132`
- 調査日: 2026-04-29
- 目的: 利用者がフォントデータを指定する方式で、`movie-telop-transcriber` の実動画 OCR 精度を現実的に改善できるか整理する。

## 現行実装の前提
- 実動画 OCR は `tools/ocr/paddle_ocr_worker.py` から `PaddleOCR.predict(image_path)` を呼び出す構成である。
- worker が受け取る主な入力は画像パス、言語ヒント、しきい値系設定、前処理設定である。
- 現行アプリでは `font_family` は未推定で `null` を許容しており、`font_size` は OCR 矩形高さ相当値である。
- 文字色や枠色は OCR 後の属性解析で推定しているが、OCR 本体はフォント情報を前提にしていない。

## 公式情報の確認結果

### 1. 推論時にフォントファイルを渡して認識精度を上げる仕組みは確認できなかった
- PaddleOCR の OCR 利用手順と推論パラメータ説明では、入力は画像パス、モデル、言語、しきい値、保存先などであり、認識精度向上用のフォント入力は見当たらない。
- 推論パラメータにある `vis_font_path` は可視化用フォントであり、認識モデルへの条件入力ではない。
- FAQ にある `PADDLE_PDX_LOCAL_FONT_FILE_PATH` も、OCR 予測結果の表示フォント変更として説明されている。

### 2. フォントは学習データ生成側では使われている
- PaddleOCR の学習資料では、多言語学習や小言語向けに「異なる corpus と fonts を使って synthetic data を生成する」と説明されている。
- 認識モデルの精度をフォントに寄せて改善したい場合、現実的な入口は「対象フォントを使った合成データ作成と再学習」であり、推論時オプション追加ではない。

### 3. 特殊フォントは一般 OCR より別問題として扱われている
- FAQ では、看板や広告の artistic fonts は難しい課題であり、辞書に載る語であれば一般 OCR ではなくテンプレートや画像検索寄りの方法を試す案が示されている。
- これは「フォントファイルを渡せばそのまま解決する」種類の問題ではない。

## このアプリで考えられる方式候補

### A. 推論時に利用者フォントを読み込ませる
- 実現性: 低い
- 理由:
  - 現行 PaddleOCR worker にその入力口がない。
  - 公式資料でも推論時フォント注入は確認できない。
  - 実装しても内部的には可視化用途に留まる可能性が高い。
- 判断: 不採用

### B. 利用者フォントを使って OCR モデルを再学習する
- 実現性: 中
- 精度改善余地:
  - 特定の番組テロップ、固定デザイン、継続運用する対象動画では改善余地がある。
  - 小書き仮名、装飾文字、似字が多いケースでは一定の改善可能性がある。
- 導入コスト:
  - 対象フォントの収集
  - 文字集合と辞書整備
  - 合成データ生成
  - 再学習環境、評価データ、配布モデル管理
  - 利用者ごとに再学習済みモデルをどう配布・切替するかの設計
- 保守コスト:
  - フォント追加のたびに再学習または再評価が必要
  - アプリ配布物と導入手順が大きく複雑化する
- 判断: 現行アプリの通常機能としては不採用

### C. OCR 後にフォント照合や補助辞書的な後処理を行う
- 実現性: 中
- 精度改善余地:
  - 効果が出るのは限定的で、主に誤認識しやすい文字の絞り込みである。
  - 例: `や/ゃ`、`つ/っ`、`ロ/口/□`、記号類、固定デザインの番組テロップ
- 制約:
  - 現行アプリは文字単位の切り出しやフォント照合基盤を持っていない。
  - 一般動画向けに広く効く方式ではなく、対象依存になりやすい。
  - UI にフォント登録機能を追加しても、利用者が期待するほど広範な改善にはつながりにくい。
- 判断: 一般機能としては見送り。必要なら将来の限定 PoC 候補。

## 精度向上が期待できる条件
- 対象動画のテロップ書体がほぼ固定である。
- 誤認識パターンが限定されており、比較対象文字も少ない。
- 利用者が対象フォントと評価用サンプルを継続的に用意できる。
- 一般 OCR 精度ではなく、特定番組向けの最適化を許容できる。

## 精度向上が期待しにくい条件
- 動画ごとにフォントや装飾が大きく異なる。
- 誤認識の主因がフォントではなく、低解像度、圧縮ノイズ、縁取り、背景干渉、モーションブラーである。
- 利用者がフォントを登録しても、再学習や辞書調整なしで即座に改善することを期待している。
- 汎用配布アプリとして、簡単な設定だけで広く改善させたい。

## 結論
- 現行 `movie-telop-transcriber` に「利用者が複数フォントデータを指定し、推論時に参照させて OCR 精度を上げる機能」を追加する方針は採用しない。
- 理由は次のとおり。
  - PaddleOCR の公式資料上、推論時フォント注入による認識精度改善の仕組みを確認できない。
  - フォントが関係する現実的な改善策は、合成データ生成と再学習、または限定的な後処理であり、現行配布アプリの導入負荷と保守負荷に見合わない。
  - 現在の優先課題は、OCR 前処理、候補フレーム選別、色推定、UI/運用改善の方が効果と実装規模のバランスが良い。

## 将来の検討余地
- 特定番組向けに誤認識パターンが明確になった場合は、フォントそのものではなく「限定文字集合に対する後処理補正」を別 Issue で PoC 化する余地はある。
- ただしその場合も、対象動画、誤認識一覧、正解データ、期待改善幅を先に固定する必要がある。
